Mainline Is Always Buildable

main represents the best known integrated state. It must remain buildable and testable at all times.

Policy

  • Must: All changes land on main via PR with required checks and reviews.
  • Must: Required CI checks must pass before merge (build + unit tests at minimum).
  • Must: If a bad merge breaks main, treat it as a priority incident: fix or revert immediately.
  • Must not: Disable failing checks to ‚Äúunblock‚Äù merges.
  • Should: Keep PRs small enough to review and rollback safely.
Rationale

A broken mainline blocks everyone, delays stabilization, and increases integration risk. Keeping main healthy is the cheapest way to reduce future release pain.

Operational expectations

When main breaks
  1. Stop merging additional PRs.
  2. Identify the offending merge.
  3. Revert or hot-fix quickly.
  4. Restore green before continuing merges.
Good hygiene
  • Use feature flags for incomplete work.
  • Prefer backward-compatible changes during active development.
  • Keep build times reasonable; flaky tests are treated as defects.
