One Repo, Multiple Environments

Our code lives in one repository. Dev/UAT/Prod are deployment targets, not separate repos and not separate ‚Äúenvironment branches‚Äù.

Applies to: All repos Goal: Prevent drift

Policy

  • Must: Use a single repository for the product/service.
  • Must: Promote the same release artifact across environments (Dev ‚Üí UAT ‚Üí Prod).
  • Must not: Create separate repositories for Dev/UAT/Prod.
  • Must not: Maintain long-lived branches named dev, uat, prod.
Rationale

Multiple repos or environment branches create silent divergence (‚Äúworks in UAT but not Prod‚Äù) and make auditing hard. A single repo + promoted artifacts ensures traceability and repeatability.

How environments differ (the right way)

Allowed differences
  • Configuration (URLs, credentials, feature flags)
  • Secrets / key material
  • Infrastructure sizing / scaling policies
  • Observability settings (sampling rates, log levels)
Not allowed differences
  • Different code for each environment
  • Different build process per environment
  • Manual patching of servers/containers in Prod

Procedure

  1. Build a Docker image from a commit/tag (immutable identifier).
  2. Deploy that same image to Dev for verification.
  3. Promote the same image to UAT for stabilization.
  4. Promote the same image to Prod for shipment.
