Release Artifacts Are Immutable

The artifact that passes UAT is the artifact that reaches Prod. For us, that artifact is a Docker image identified by an immutable tag (SHA / version) or digest.

Policy

  • Must: Build release artifacts from a specific commit or tag.
  • Must: Use immutable identification: image digest and/or git SHA, plus optional version tag vX.Y.Z.
  • Must: Promote the same artifact across environments (no rebuilds per environment).
  • Must not: Deploy ‚Äúlatest‚Äù tags to UAT/Prod.
  • Must not: Patch running containers/servers in Prod.
Rationale

Rebuilding between UAT and Prod introduces drift (different dependency resolution, base image changes, build metadata). Immutability ensures repeatability, traceability, and reliable rollback.

Recommended Docker tagging strategy

Tag type Example Mutable? Use
Commit SHA tag myapp:sha-abc1234 No Internal promotion & traceability
Release tag myapp:v2.4.0 No Customer / Prod reference
Digest pin myapp@sha256:... No Strongest immutability for deploy manifests
Moving env tag myapp:uat Yes Optional convenience only (never the source of truth)

Procedure

  1. CI builds myapp:sha-<commit> from the commit and pushes to registry.
  2. UAT deploy references the SHA tag (or digest).
  3. Shipping creates git tag vX.Y.Z and maps it to the same image digest.
  4. Prod deploy references myapp:vX.Y.Z (or digest).
