Squash vs Rebase Merge

Choose the merge strategy that keeps history readable and traceable while protecting protected branches from unsafe history rewriting.

Default stance

Squash merge (default)

Use for feature branches into main and release/*. It keeps one commit per PR, simplifies rollback, and reduces noise from WIP commits.

Rebase merge (opt-in)

Allow only when the branch already has a clean, reviewable history and rebasing will not rewrite shared commits. Apply to keep linear history on small, high-discipline teams.

Rules of engagement

  • Squash is the default on all repositories. Teams should document any deviation.
  • Do not rebase protected branches (main, release/*, tags). History must remain immutable for traceability.
  • Rebase merge requires a clean branch: no merge commits, up-to-date with base, and approvals still valid after rebase.
  • Each PR produces one logical commit in history. Curate commit messages to reference tickets and describe the change.

Operational notes

Rollback friendliness

Squash merges make git revert predictable. One revert equals one PR.

Release hygiene

Use squash on release/* to preserve a clean audit trail from tag to commits. Avoid merge commits that obscure the shipped diff.
