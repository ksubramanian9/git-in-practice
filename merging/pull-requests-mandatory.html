PRs Are Mandatory

Every change merges through a reviewed pull request with passing checks. This preserves a buildable mainline, auditable history, and alignment with release traceability.

Policy

  • Must: All merges into protected branches (main, release/*, hotfix/*) go through a PR.
  • Must: Required CI checks (build, tests, linters) pass before merge; disablement is forbidden.
  • Must: At least one CODEOWNER or designated reviewer approves the PR; self-approval alone is not sufficient for shared code.
  • Must: PRs reference work items or release tickets for traceability.
  • Should: Keep PRs small and focused to reduce review latency and rollback cost.
  • Must not: Push directly to protected branches or bypass review using admin privileges.
Rationale

PR gates are how we keep main buildable, releases traceable, and production recoverable. They create reviewable checkpoints, surface risks early, and provide an audit trail for shipped artifacts.

Pull request lifecycle

Author responsibilities
  1. Branch from the correct base (main or the active release/*).
  2. Open the PR early with a concise title, summary, and linked ticket.
  3. Keep the branch current (rebase or merge main) before requesting final review.
  4. Ensure required checks pass; rerun flaky checks rather than merging red.
  5. Address review feedback promptly and transparently.
Reviewer responsibilities
  • Confirm scope: correct base branch, minimal change surface, clear rollback strategy.
  • Verify tests and required checks are green; request evidence if missing.
  • Inspect risk areas: migrations, feature flags, backward compatibility.
  • Block merges that endanger main stability or traceability.

Quality bar

  • Definition of ready
    • Scoped to a single concern; breaking changes behind flags.
    • Includes tests or an explicit rationale for missing coverage.
    • Describes user impact and operational risks.
  • Definition of done
    • Required reviews complete; comments resolved.
    • All checks green; no force merges of red pipelines.
    • Deployment notes captured (feature flags, runbooks, migration steps).

Exception handling

Emergencies are rare. When production needs an immediate fix, we still enforce traceability.

  • Use a hotfix/* branch from the production tag and open a PR with highest priority.
  • Require at least one reviewer; pair review in chat or call if time constrained.
  • After merge, tag the release, deploy, and back-merge to main to keep history linear.
  • Record the incident and justification in the release log.
