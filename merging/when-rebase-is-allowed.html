When Rebase Is Allowed

Rebasing is powerful but risky. Use it only on private or short-lived branches to maintain linear history without rewriting published commits.

Allowable cases

  • Keeping a feature branch current: Rebase onto main before requesting review to reduce merge noise.
  • Commit cleanup before opening a PR: Squash local WIP commits into logical units before the branch is shared.
  • Syncing forks: Rebasing a fork on top of upstream to avoid diverging merge commits when required by upstream policy.
  • Small doc-only PRs: With one or two commits, rebasing for linear history is acceptable if no one else has pulled the branch.

Hard prohibitions

Never rebase
  • main, release/*, hotfix/*, or any branch referenced by a production tag.
  • Branches with commits already consumed by others (shared branches).
  • After a PR is approved unless reviewers re-acknowledge the rebased commits.
Reasoning

Rewriting public history breaks traceability, invalidates reviews, and can ship unreviewed code if tags move. It also conflicts with immutable release artifacts.

Safe rebase checklist

  1. Confirm no one else depends on the branch (not shared, no open PR approvals).
  2. Fetch latest main or target branch: git fetch origin.
  3. Run git rebase origin/main (or the correct base).
  4. Resolve conflicts locally; rerun tests to keep the branch buildable.
  5. Force-push the rebased branch with git push --force-with-lease to avoid clobbering others.
  6. Re-request review if commits changed since approval.

Alternatives to rebasing

Merge main into the branch

Safe for shared branches; preserves history and avoids rewriting approvals. Clean up with squash at merge time.

Use feature flags

Ship incremental changes behind flags instead of long-lived branches that need repeated rebases.
