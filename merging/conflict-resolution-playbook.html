Conflict Resolution Playbook

Resolve merge conflicts quickly without sacrificing stability or traceability. Keep mainline buildable and minimize risk during reconciliation.

First principles

  • Protect mainline: Never merge conflict resolutions that break builds or tests. Verify locally.
  • Prefer re-apply over guesswork: Understand intent of both sides; avoid deleting changes you do not own.
  • Keep history explainable: Resolve conflicts on the branch, then merge via PR; avoid force pushes to protected branches.

Resolution steps

  1. Fetch latest base: git fetch origin.
  2. Update branch with the base (git merge origin/main or git rebase origin/main if allowed).
  3. Open conflicted files and inspect both sides. Keep intent from each branch; re-run code generation where applicable.
  4. Run tests and linting locally; for risky areas, add quick unit or integration checks to validate the merge.
  5. Push the resolved branch (use --force-with-lease if rebasing).
  6. Request re-review from affected owners before merging.

Triage cheat sheet

Configuration files
  • Preserve required CI steps and version pins.
  • Prefer the stricter setting when in doubt.
Migrations & schemas
  • Ensure ordering is correct; avoid reusing migration IDs.
  • Prefer additive changes; rerun migration generators after resolving.
Front-end assets
  • Rebuild bundles if package versions changed.
  • Check snapshots or visual diffs to avoid regressions.

Escalation and safety nets

  • When unsure, pair with a domain owner; do not guess.
  • If conflicts are due to risky refactors, consider reverting one side and re-applying via a follow-up PR.
  • For release branches, prefer cherry-picking known-good commits over manual conflict editing.
  • Document the resolution in the PR description so reviewers know what changed.
