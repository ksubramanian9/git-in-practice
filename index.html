Git in Practice Branching, Releases, and Team Operating Model

This is an internal operating manual: how our teams use Git to collaborate, stabilize versioned releases, and ship Docker images with traceability across Dev/UAT/Prod.

Operating Model (in 8 lines)

  1. All development merges into main via PR with required CI + review.
  2. When UAT starts, cut release/x.y from main.
  3. Only bug fixes are merged into release/x.y during stabilization.
  4. Build Docker image tagged by SHA (and later vX.Y.Z tag).
  5. Promote the same image across Dev â†’ UAT â†’ Prod.
  6. Ship by creating immutable tag vX.Y.Z on release branch.
  7. Merge release branch back to main, then delete it.
  8. Hotfixes are branched from the production tag and forward-ported to main.
