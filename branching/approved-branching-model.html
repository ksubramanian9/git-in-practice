Approved Branching Model Latest-only releases with long UAT stabilization

This page defines the standard branching model for repositories that ship versioned customer releases, support only the latest version, and require a multi-day UAT window. The goal is to keep UAT stable without freezing all development work.

Status: Draft Applies to: Latest-only products Release artifact: Docker image

Model Summary

Branches we use
  • main ‚Äî continuous development; must remain buildable
  • release/x.y ‚Äî short-lived stabilization branch during UAT
  • feature/* ‚Äî short-lived development branches off main
  • fix/* ‚Äî short-lived bugfix branches off release/x.y during UAT
  • hotfix/* ‚Äî emergency patch branch from a production tag (rare)
The one-line rule

After UAT begins, features continue on main, while only bug fixes go into release/x.y until the version is tagged and shipped.

We avoid repo-per-environment and environment branches; environments are deployment targets.

Policy

Must: All development work merges into main via PR; direct pushes to main are not allowed.

Must: When UAT starts, create release/x.y from a chosen green commit on main (the ‚Äúrelease cut‚Äù).

Must: During stabilization, release/x.y accepts only bug fixes and release-only changes (e.g., version metadata).

Must: Production/customer shipment is always tied to an immutable tag (e.g., vX.Y.Z).

Must: Build Docker images with immutable identifiers (commit SHA and/or version tag). Promote the same image through environments.

Must not: Create separate repositories for Dev/UAT/Prod or ‚Äúpromote code‚Äù by copying between repos.

Must not: Merge new features into release/x.y after the release cut.

Rationale

This model keeps UAT stable (controlled change surface) while allowing parallel feature development on main. It also preserves traceability: ‚ÄúProd is running tag vX.Y.Z (commit SHA ‚Ä¶)‚Äù.

Procedure

A) Normal development (default state)
  1. Create feature/* from main.
  2. Open PR ‚Üí required CI checks ‚Üí required review ‚Üí merge to main.
  3. Main remains green and continuously integrable.
B) UAT stabilization (release cut)
  1. Cut release/x.y from a selected green commit on main.
  2. Deploy UAT only from release/x.y (not from main).
  3. Bug fixes branch from release/x.y as fix/* and merge back into release/x.y.
C) Ship (tag and release)
  1. Freeze changes into release/x.y (tighten approvals).
  2. Create tag vX.Y.Z on the exact commit being shipped.
  3. Build (or re-tag) Docker image and distribute to customers.
D) Close release (mandatory hygiene)
  1. Merge release/x.y back into main (forward the fixes).
  2. Delete release/x.y after shipment (latest-only).
  3. Record release notes and the shipped tag in the release log.
Reference commands (example)
git fetch origin
git checkout main
git pull
git checkout -b release/2.4
git push -u origin release/2.4

# later, ship
git checkout release/2.4
git tag v2.4.0
git push origin v2.4.0

Note: enforce branch protections so release cuts and tags are performed only by authorized roles.

Branch Topology

A simplified view of how main and release/x.y diverge during UAT, then converge after shipping. 

main:      A ‚Äî B ‚Äî C ‚Äî D ‚Äî E ‚Äî F ‚Äî G  (features continue)
                 \
release/2.4:       C ‚Äî c1 ‚Äî c2 ‚Äî [TAG v2.4.0] ‚Äî (merge back to main)

Legend:
- C : release cut (branch created)
- c1,c2 : UAT bug fixes only
- TAG : customer shipment point

Exceptions

Hotfix allowed when a customer-impacting production defect cannot wait for the next planned release.

  • Create hotfix/* from the production tag (e.g., v2.4.0).
  • Tag the patch release (e.g., v2.4.1).
  • Merge hotfix back into main immediately after shipment.

All exceptions must be recorded in the release log and approved by the repo owners (or Platform).
