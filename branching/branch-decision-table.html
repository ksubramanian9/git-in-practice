Branch Decision Table

A single reference table: if X happens, which branch to create, from where, where to merge, which environments are affected, and what extra rules apply.

Applies to: Latest-only releases UAT window: Multi-day Artifact: Docker image
Quick rule: Features branch from main. During UAT, bugfixes branch from release/x.y. Production emergencies branch from the production tag.

Decision Table

Scenario Create branch Branch from Merge into Deploy target Notes / extra rules
New feature work (default) feature/<short-desc> main main Dev / CI preview PR required. Keep branches short-lived.
Bug found during normal development fix/<issue-id>-<short-desc> main main Dev / CI preview Treat as normal PR. Prefer adding a test that fails before the fix.
Start UAT stabilization (release cut) release/x.y main (green commit) N/A (later merge back) UAT from release/x.y After cut: no new features into release branch.
Bug found in UAT after release cut fix/<issue-id>-<short-desc> release/x.y release/x.y UAT Fix must be minimal. Add test if possible. Do not merge unrelated refactors.
Same UAT bug also exists in main fix/<issue-id>-forwardport main main Dev / CI After stabilizing, either (a) merge release back to main, or (b) do a small forward-port PR. Choose the simpler path that avoids conflicts.
Ship customer release (no new branch) create tag release/x.y Tag vX.Y.Z Prod / Customer Prod deploy must reference a tag (or digest). Build once; promote artifact.
Close release (post-ship) (merge activity) release/x.y main N/A Merge release back to main. Delete release branch (latest-only).
Production emergency bug (hotfix) hotfix/<issue-id>-<short-desc> Tag vX.Y.Z currently in prod Tag vX.Y.(Z+1) then merge to main Prod Keep change minimal. Forward-port to main is mandatory to prevent regression.
Experimental spike / POC feature/spike-<short-desc> main main (optional) None / ephemeral Prefer a draft PR. If not merged, branch must be deleted.

Guardrails (Common Mistakes We Avoid)

  • Do not create long-lived dev, uat, prod branches.
  • Do not merge feature work into release/x.y after the release cut.
  • Do not ‚Äúpromote code‚Äù by copying between repositories.
  • Prefer shipping the same Docker image (digest) across environments to avoid rebuild drift.
