Branch Types & Naming Conventions

This page defines the only allowed branch types and the mandatory naming conventions for repositories following the approved branching model. These conventions enable automation, reduce cognitive load, and make repository state obvious to everyone.

Why Branch Naming Is a Policy (Not Style)

Branch names are not cosmetic. They encode intent, lifecycle, and risk. CI pipelines, environment rules, and permissions depend on predictable naming.

Design principle: From the branch name alone, an engineer should know:
  • Why the branch exists
  • Where it may be merged
  • How long it should live
  • What approvals apply

Naming Rules (Mandatory)

  • Must: Use lowercase letters only.
  • Must: Use hyphens for word separation.
  • Must: Keep names short, descriptive, and task-oriented.
  • Must: Include issue or ticket ID when available.
  • Must not: Encode environment names (uat, prod, dev) in branch names.
  • Must not: Create long-lived personal branches.

Examples

Good
  • feature/user-login-rate-limit
  • fix/JIRA-412-null-pointer
  • release/2.4
  • hotfix/INC-7782-token-expiry
Bad
  • feature123
  • john-dev-branch
  • uat-fixes
  • release-final-final

Enforcement

These conventions are enforced via:

  • Branch protection rules (GitHub / GitLab)
  • CI validation on branch name patterns
  • PR templates and CODEOWNERS
