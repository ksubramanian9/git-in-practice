Release Lifecycle

How we plan, cut, stabilize, and ship releases so every environment reflects a deliberate state with clear ownership and traceability.

Policy

  • Must: Every release begins from main after required checks are green and scope is declared.
  • Must: Release branches are short-lived and created only for stabilization and UAT hardening.
  • Must: After the cut, release/x.y accepts only stabilization fixes; new features continue on main.
  • Must: Each stage (cut â†’ harden â†’ tag â†’ deploy) has explicit exit criteria owned by the release manager.
  • Should: Freeze risky changes after the cut; defer scope creep to the next release unless approved by incident command.
  • Must not: Cherry-pick unreviewed commits; all fixes flow through PRs with the same quality gates as main.
Why this matters

A predictable lifecycle reduces surprises for QA, keeps production deployable at will, and makes rollback strategies straightforward.

Stages & gates

Cut
  • Create release/x.y from main after CI is green and scope is confirmed.
  • Enable release branch protections (PRs required, status checks, CODEOWNERS).
  • Record owner, scope, and planned date in the release notes draft.
Harden
  • Stabilization happens on the release branch; only bug fixes and safety switches allowed.
  • Automated smoke, regression, and security scans must be green.
  • Release notes updated for each fix (issue ID + risk + test evidence).
Tag & sign-off
  • Tag the release branch HEAD with a semantic version (see Versioning & Tags) once exit criteria are met.
  • Publish release notes with commit range, known issues, and rollout plan.
  • Release manager signs off and hands over to deployment owners.
Deploy & monitor
  • Promote the tagged artifact through environments (Dev â†’ UAT â†’ Prod) without rebuilds.
  • Verify health metrics and error budgets after each hop; pause if SLOs degrade.
  • Finalize post-release review with lessons learned and follow-up tasks.

Roles & responsibilities

Release manager
  • Owns the calendar, scope, and cut decision.
  • Approves exceptions and coordinates sign-off.
  • Ensures documentation and tags are published.
QA lead
  • Defines the test matrix and exit criteria.
  • Tracks defects and communicates risk.
  • Signs off UAT readiness.
Engineering owner
  • Executes fixes via PR, keeps branch green.
  • Maintains backward compatibility during rollout.
  • Prepares rollback plan and feature flag toggles.
