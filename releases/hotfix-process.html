Hotfix Process

How to ship urgent fixes without destabilizing mainline development or violating release traceability.

Policy

  • Must: Declare an incident commander for any production-impacting hotfix.
  • Must: Branch from the current production tag (or its commit), not from main.
  • Must: Use a dedicated hotfix/x.y.z-n branch with the same protections as release branches.
  • Should: Keep changes minimal; prefer feature flags or configuration toggles to reduce blast radius.
  • Must not: Bypass PR review or required checks; emergency does not remove quality gates.

Execution steps

  1. Identify the production tag to fix and branch hotfix/x.y.z-1 from that commit.
  2. Implement the minimal fix, add targeted tests, and open a PR to the hotfix branch.
  3. Run mandatory checks (unit, smoke, security) and capture evidence in the PR.
  4. Tag the hotfix branch HEAD as vX.Y.Z+hotfixN once approved and green.
  5. Promote the tagged artifact directly to the affected environments with heightened monitoring.
  6. Backport the hotfix to main (and active release branches) via a follow-up PR to prevent regression.
Communication

Announce start/end of the hotfix window in #eng-ops with the tag, owners, and risk summary. Update the incident ticket with rollout timing and monitoring status.

Quality & rollback

Quality controls
  • Automated checks must stay enabled; flaky tests are triaged, not disabled.
  • Include reproduction steps and validation checklist in the PR description.
  • Security review required for fixes touching auth, data access, or secrets.
Rollback readiness
  • Keep the previous production tag available for immediate redeploy.
  • Document any data migrations or feature flag changes that must be reverted.
  • Monitor error rates, latency, and business KPIs for at least one full cycle post-deploy.
