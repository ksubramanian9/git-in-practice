Versioning & Tags

Standardized semantic versions, annotated tags, and build metadata that let us trace any deployment back to source.

Versioning rules

  • Must: Use MAJOR.MINOR.PATCH (SemVer) with optional pre-release (-rc.n) and build metadata (+sha).
  • Must: Bump MAJOR when backward-incompatible; MINOR for additive, backward-compatible changes; PATCH for fixes.
  • Must: Pre-release identifiers (-rc, -beta) are only cut from release branches, never from feature branches.
  • Should: Include the short commit SHA in build metadata for container images (1.4.0+abc1234).
  • Must not: Reuse or move a tag once published. Treat tags as immutable release artifacts.

Tagging procedure

  1. Confirm the release branch is green and release notes are ready.
  2. Create an annotated tag (git tag -a vX.Y.Z -m "Release X.Y.Z") on the release branch HEAD.
  3. Push the tag (git push origin vX.Y.Z) and verify CI publishes artifacts that reference the tag.
  4. Record the commit SHA, tag name, and artifact digests in the release notes.
  5. Promote the tagged artifact through environments without rebuilding.
Annotation checklist
  • Version number and codename (if any)
  • Commit SHA and compare link to previous tag
  • Summary of notable changes and risks
  • Build artifact digests (container image, package hash)

Mapping versions to deployments

Containers
  • Images are tagged with vX.Y.Z and vX.Y.Z+sha.
  • Runtime deployments reference the immutable digest, not latest.
  • Promotions reuse the same digest across Dev/UAT/Prod.
Infrastructure & docs
  • Helm charts and manifests pin the release tag; configuration diffs are tracked per environment.
  • Docs and runbooks reference tags when describing behavior or mitigations.
  • Rollback plans specify the previous stable tag and required data migrations (if any).
