Docker: Build Once, Promote

Build a Docker image exactly once from a trusted commit, sign it, and promote that immutable image through every environment. Rebuilding per environment invites drift and eliminates traceability.

Applies to: Containerized services Goal: Artifact integrity

Build and promotion rules

  • Must: Build images from mainline or a release tag; never from local workstations.
  • Must: Tag images with immutable identifiers (commit SHA + semver tag) and push to the registry once.
  • Must: Promote the exact image digest between Dev, QA/UAT, and Prod via CI/CD pipelines.
  • Must not: Rebuild images for each environment or manually retag images in registries.

Pipeline blueprint

Build & sign
  1. Checkout commit/tag, build once, and run security scanning.
  2. Sign the image (cosign/notary) and push to the registry.
  3. Publish a provenance attestation (SBOM + digest).
Promote & verify
  1. Deploy by digest to Dev, then QA/UAT, then Prod.
  2. Verify signatures and policy (e.g., via admission controller) before rollout.
  3. Record release notes referencing the digest for traceability.

Operational checklist

  • Parameterize environment differences via Helm/compose values, not separate Dockerfiles.
  • Keep build caches reproducible (locked base images, pinned dependencies).
  • Alert if a deployment references an image tag without a matching digest pin.
  • Use rollout strategies (blue/green or canary) to lower blast radius while promoting.
