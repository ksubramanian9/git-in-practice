Deploy Targets, Not Repos

Environments are destinations for the same codebase and the same build artifacts. We do not fork repositories or maintain environment-specific branches to represent Dev, QA, or Prod.

Applies to: All products Goal: Eliminate drift

Policy

  • Must: Treat environments (Dev, QA/UAT, Prod) as deploy targets fed by the same mainline branch.
  • Must: Promote a single build artifact through every environment without rebuilding it.
  • Must: Keep infrastructure and configuration differences in code (Terraform, Helm, env vars), not in divergent repos.
  • Must not: Create separate repos or long-lived branches per environment to "hold" different code.
  • Must not: Patch production directly; changes start from mainline and flow forward.

Anti-patterns we are avoiding

Separate environment repos

Each repo drifts differently, doubling review and release overhead.

Environment-only branches

Branches like dev and prod quietly diverge and hide hotfixes.

How to work

  1. Commit to mainline and produce an immutable artifact (container image, binary, package).
  2. Use deployment pipelines to promote that artifact through Dev â†’ QA/UAT â†’ Prod.
  3. Track configuration per environment in code, reviewed via PRs, and applied by automation.
  4. When issues occur, fix forward on mainline and rebuild/promote; never patch Prod directly.
