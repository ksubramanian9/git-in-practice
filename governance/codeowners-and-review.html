CODEOWNERS & Review Enforce the right reviewers for every change

CODEOWNERS files route pull requests to the people accountable for quality. This page standardizes how we define ownership, structure review rules, and keep CODEOWNERS maintainable as the codebase grows.

Policy

Must: Every repository contains a CODEOWNERS file at the repo root; monorepos add team-owned entries per folder.

Must: Default ownership points to the platform team; more specific patterns override defaults for product areas and shared libraries.

Must: PRs require at least one approval from a matching CODEOWNER; security-sensitive paths can require 2 approvals using branch protection.

Should: List teams (e.g., @org/team-abc) instead of individuals so vacations do not block merges.

Should: Include a short comment block at the top explaining how to request ownership changes.

Must not: Add wildcard owners that bypass domain experts (e.g., * @everyone).

File Layout

Recommended structure
  • Header comment describing escalation path.
  • Product domains grouped by folder (e.g., api/, ui/, platform/).
  • Shared libraries explicitly owned (logging, auth, design system).
  • Catch-all default owner at the bottom.
  • Temporary overrides expire with a date and issue link.
Sample snippet
# Owners: add/edit via Platform ticket
/ui/              @org/frontend
/api/             @org/backend
/platform/ci/     @org/devex
*                 @org/platform

Use team aliases so reviews stay resilient to staffing changes.

Operational Guidance

Submission: CODEOWNERS updates follow the same PR standards as code (CI + review from existing owners).

Audits: Quarterly review CODEOWNERS entries with team leads; remove deprecated teams and confirm coverage for new domains.

Escalation: If owners are unresponsive after 2 business days, Engineering Managers can apply a one-time override with a documented reason.

Tooling: Enable branch protection to require CODEOWNER review and dismiss stale approvals after force-push or file changes.

Rationale

CODEOWNERS keeps expertise aligned with code paths and prevents high-risk areas from being merged without proper oversight. Standard structure avoids drift between teams and speeds up onboarding.
